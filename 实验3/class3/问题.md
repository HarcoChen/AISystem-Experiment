Numworker
2.这个报错里最关键的不是你的 search space，而是这句：

```python
params = self._running_params.pop(parameter_id)
```

NNI 的 TPE 在收到 trial 最终结果时，会从 `_running_params` 里把这个 `parameter_id` 弹出去；而 `trial_end()` 也会把它从同一个表里移除。于是出现 `KeyError: 0` 的含义就是：**NNI 收到这个 trial 的 final metric 时，它已经不再认为这个 parameter_id 还“在运行中”了**。换句话说，就是“结果到得太晚 / 到了两次 / trial 已经先被结束了”。([NNI Documentation][1])

先说结论：**这不是你现在这个嵌套 search space 写法本身的问题。** NNI 的 stable 文档明确写了，`choice` 里嵌套子搜索空间、并用 `_name` 区分分支，是合法写法；而且 **TPE** 本来就是支持 nested search space 的。([NNI Documentation][2])

更像的原因有两个。

第一，你现在是 **TPE + local + `trialConcurrency: 3`**。这个组合在 NNI 里确实有人报过**几乎同样的错误**：issue #5689 里就是 `trialConcurrency > 1` 时，TPE 在 local mode 下抛出同样的 `params = self._running_params.pop(parameter_id)` / `KeyError`；而 2024 年又有 #5798、#5802 两个 open issue，都是同一路径崩掉。NNI 的 GitHub 仓库还在 2024-09-18 被归档成只读，所以这些 open issue 至少在那个仓库里没有后续修复记录。([GitHub][3])

第二，如果你最近把 `main.py` 和 `train.py` 分开后，是用 `subprocess`、`multiprocessing` 或再起一个 Python 进程去跑 `train.py`，那也很容易触发这种问题。NNI 文档要求每个 trial 的 `get_next_parameter()` 只调用一次，并在 **trial 结束前** 用 `report_final_result()` 上报结果。官方 CIFAR-10 示例也是把 `nni.get_next_parameter()`、`nni.report_intermediate_result()`、`nni.report_final_result()` 都放在同一个 `main.py` trial 里完成。要是你让子进程也碰 NNI API，或者父进程先结束、子进程后上报，就很容易出现“trial 先结束，结果后到”的状态错位。([NNI Documentation][4])

我建议你直接这么处理。

先把 **TPE 的并发降到 1**：

```yaml
trialConcurrency: 1
tuner:
  name: TPE
  classArgs:
    optimize_mode: maximize
```

因为你现在看到的这个报错，和已知的 TPE/local/并发问题太像了。等 TPE 单并发能稳定跑通，再考虑别的。([GitHub][3])

然后把代码结构收成下面这种：**只有 `main.py` 接触 NNI，`train.py` 只负责训练并返回分数。**

`main.py`：

```python
import nni
from train import train_once

def main():
    params = {
        'batch_size': 128,
        'num_conv_layers': 3,
        'dropout_rate': 0.5,
        'optimizer': {
            '_name': 'Adam',
            'lr': 1e-3,
            'weight_decay': 1e-4
        }
    }
    nni_params = nni.get_next_parameter()
    params.update(nni_params)

    best_acc = train_once(params)   # 普通函数调用，不要再起一个带 nni 的子进程
    nni.report_final_result(best_acc)

if __name__ == "__main__":
    main()
```


如果你现在是这样写的：

```python
subprocess.run(["python", "train.py"])
```

那我建议先不要这么跑。至少在定位这个问题时，先改成 `from train import train_once` 的直接函数调用，避免父子进程各自持有 NNI 上下文。

再稳一点的话，你可以做一个最小排查：

1. `trialConcurrency` 改成 1。
2. `main.py` 保证只有一次 `nni.get_next_parameter()`。
3. `nni.report_final_result()` 只在 `main.py` 里调用一次。
4. `train.py` 不 import `nni`。
5. 先把 `report_intermediate_result()` 也临时去掉，只保留 final result，确认是否还报错。

如果这样还能复现，那我会更倾向于是 **NNI 3.0 的 TPE/local bug**，而不是你的代码逻辑。因为公开 issue 里已经有多个非常接近的案例了。([GitHub][3])
