# 实验四：分布式训练任务清单

## 1. 实验目的与环境准备
- [ ] **理解核心概念**：深入理解数据并行（Data Parallel）与模型并行（Model Parallel）的思想。
- [ ] **硬件检查**：确认多核 CPU 及 NVIDIA GPU（推荐 2 块及以上，支持多 GPU 并行）。
- [ ] **软件环境**：确保已安装 Python、PyTorch（含 `torch.distributed`）以及对应的 CUDA 环境。

## 2. 分布式训练基础实现 (5.4.1)
- [ ] **基准程序准备**：选择一个规模较小的 PyTorch 模型，确保单机单 GPU 训练流程正常。
- [ ] **DDP 模式改造**：
    - [ ] 初始化进程组：`torch.distributed.init_process_group`。
    - [ ] 数据分发：使用 `DistributedSampler` 包装数据集。
    - [ ] 模型包装：使用 `DistributedDataParallel` (DDP) 包装模型。
- [ ] **多 GPU 验证**：运行程序并观察不同进程输出，确保 `rank` 和 `world size` 配置正确，训练无报错。

## 3. GPU 数量对训练性能的影响分析 (5.4.2)
- [ ] **对比实验**：保持其他参数一致，分别在 **1 张** 和 **2 张**（或更多）GPU 下运行。
- [ ] **指标记录**：记录单轮训练耗时、总训练时间、吞吐率 (samples/sec)。
- [ ] **数据计算**：计算加速比（Speedup），并与理想线性加速情况对比。
- [ ] **性能分析**：分析为何无法达到 100% 线性加速，探讨通信与同步带来的额外开销。

## 4. 批量大小 (Batch Size) 对通信开销的影响 (5.4.3)
- [ ] **固定变量**：固定 GPU 数量和模型结构。
- [ ] **参数扫描**：设置不同的 `batch_size`（如 32, 64, 128, 256 等）。
- [ ] **量化通信开销**：
    - [ ] 测量单次迭代总时间（含通信）。
    - [ ] 测量纯计算时间（通过 `model.no_sync()` 或关闭梯度同步近似获得）。
    - [ ] 计算 `通信时间 ≈ 总时间 - 纯计算时间`。
- [ ] **趋势总结**：分析通信占比随 `batch_size` 变化的规律（通常随 batch size 增大而占比下降）。

## 5. 模型参数规模对通信开销的影响 (5.4.4)
- [ ] **选型对比**：选择一个小模型和一个参数量较大的模型。
- [ ] **重复实验**：在相同条件下测量两者的通信时间占比。
- [ ] **瓶颈探讨**：分析模型规模如何影响通信量，探讨在大模型训练中通信是否成为主要瓶颈。

## 6. 选做实验 (5.4.5 & 5.4.6)
- [ ] **后端对比**：在相同环境下对比 `Gloo` 与 `NCCL` 后端的性能差异。
- [ ] **多机部署**：尝试在多台机器搭建分布式环境，配置 `MASTER_ADDR`、`MASTER_PORT` 及 SSH 免密登录。

## 7. 实验报告撰写要求 (5.5)
- [ ] **环境说明**：记录完整的硬件参数（CPU/GPU/显存）和软件版本信息。
- [ ] **结果图表**：设计数据表格，整理并展示不同配置下的时间、吞吐率及通信占比。
- [ ] **总结讨论**：总结实验规律，深入理解分布式机制，并调研 **FSDP**、**DeepSpeed** 等主流大模型分布式系统现状。
