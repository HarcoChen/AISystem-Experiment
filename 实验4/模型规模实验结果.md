# 模型参数规模对通信开销的影响

## 实验设置

- 数据集：CIFAR-10
- GPU 数量：2
- 每个进程 batch size：128
- 训练轮数：5 epoch
- 分布式方式：PyTorch DDP
- 通信后端：NCCL

本实验固定 GPU 数量和 batch size，只改变模型规模，对比不同模型下的训练时间、吞吐率和通信开销占比。

## 汇总表

| 模型 | 参数量 | GPU 数量 | Batch Size | 总训练时间/s | 稳定吞吐率 samples/s | 后 4 轮平均通信占比 |

后 4 轮平均通信占比计算：

```text
ResNet18: (10.82% + 14.72% + 13.18% + 13.12%) / 4 = 12.96%
ResNet50: (19.08% + 19.08% + 19.88% + 20.41%) / 4 = 19.61%
```

## 结果分析

从参数量看，ResNet18 约有 1118 万个参数，ResNet50 约有 2353 万个参数，后者参数量大约是前者的 2.1 倍。按一般理解，模型参数更多时，梯度同步的数据量也会更大，因此通信压力通常会更高。

从训练时间和吞吐率看，ResNet18 的总训练时间更短，稳定吞吐率更高；ResNet50 的总训练时间更长，吞吐率更低。这说明更大的模型会带来更多计算开销，因此整体训练速度下降。

从当前实验测得的通信占比看，ResNet18 的后 4 轮平均通信占比约为 12.96%，ResNet50 约为 19.61%。这一结果更符合“大模型参数更多，通信压力更明显”的直觉：ResNet50 参数量更大，反向传播时需要同步的梯度数据更多，因此通信占比也更高。

因此，本实验可以得到较明确的结论：模型参数规模增大后，训练时间会明显增加，吞吐率下降，同时通信开销占比也会上升。随着参数规模继续增加，梯度同步带来的通信开销有可能成为大模型分布式训练中的重要性能瓶颈。
