# 通信后端对训练性能的影响

## 实验设置

- 数据集：CIFAR-10
- 模型：ResNet50
- GPU 数量：2
- 每个进程 batch size：128
- 训练轮数：5 epoch
- 分布式方式：PyTorch DDP

本实验固定模型、GPU 数量和 batch size，只改变通信后端，对比 `NCCL` 与 `Gloo` 在多 GPU 训练中的性能差异。

## 汇总表

| 后端 | GPU 数量 | 模型 | Batch Size | 总训练时间/s | 稳定吞吐率 samples/s | 后 4 轮平均通信占比 |
|---|---:|---|---:|---:|---:|---:|
| NCCL | 2 | ResNet50 | 128 | 55.6765 | 约 5330 | 19.61% |
| Gloo | 2 | ResNet50 | 128 | 110.1818 | 约 2525 | 57.25% |

后 4 轮平均通信占比计算：

```text
NCCL: (19.08% + 19.08% + 19.88% + 20.41%) / 4 = 19.61%
Gloo: (54.19% + 57.70% + 57.47% + 59.64%) / 4 = 57.25%
```

## 结果分析

从总训练时间和吞吐率看，NCCL 明显优于 Gloo。在相同实验设置下，NCCL 的总训练时间为 55.6765s，而 Gloo 为 110.1818s，几乎增加到 2 倍；稳定吞吐率方面，NCCL 约为 5330 samples/s，而 Gloo 只有约 2525 samples/s。

从通信开销占比看，NCCL 的后 4 轮平均通信占比约为 19.61%，而 Gloo 达到 57.25%，差异非常明显。这说明在 GPU 多卡训练场景下，NCCL 对 GPU 间通信做了更有针对性的优化，而 Gloo 更适合 CPU 或通用分布式通信场景，在 GPU 训练下性能明显较弱。

从准确率看，两种后端在 5 个 epoch 后的 best accuracy 接近，NCCL 为 38.95%，Gloo 为 38.01%。这说明后端主要影响的是训练效率和通信开销，而不会直接改变模型的优化目标与最终收敛方向。

因此，本实验可以得到较明确的结论：在基于 GPU 的分布式训练任务中，NCCL 比 Gloo 更适合作为通信后端。对于需要高吞吐和低通信开销的多 GPU 训练任务，应优先选择 NCCL。
